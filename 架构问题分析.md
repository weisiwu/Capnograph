# CapnoGraph 架构问题分析

日期：2026-06-14

范围：项目根目录、Android `apps/android`、iOS `apps/ios`、构建脚本、context 文档。本文只总结当前架构风险，不修改业务行为。

## 总结

当前主要问题不是单个模块实现错误，而是核心业务边界被全局状态、平台平行实现、生命周期耦合和缺少测试共同拉穿。Android 风险高于 iOS：BLE、PDF、Room、备份、打印、权限和 UI 状态耦合在少数大类里；iOS 也存在 BluetoothManager/AppConfigManage 过大、协议和 PDF 逻辑与 Android 重复实现的问题。

## 高风险问题

### 1. Android 核心职责过度集中

证据：

- `apps/android/app/src/main/java/com/wldmedical/capnoeasy/kits/BlueToothKit.kt` 同时承担扫描、连接、GATT、协议解析、状态更新、报警、设备信息查询。
- `apps/android/app/src/main/java/com/wldmedical/capnoeasy/kits/PDFKit.kt` 将 PDF 数据准备、分页、绘图、文件输出、模板配置集中在 `SaveChartToPdfTask`。
- `apps/android/app/src/main/java/com/wldmedical/capnoeasy/kits/LocalStorageKit.kt` 同时处理 Room、SharedPreferences、记录查询、PDF 触发。
- `apps/android/app/src/main/java/com/wldmedical/capnoeasy/models/AppStateModel.kt` 聚合导航、弹窗、患者、记录、设备、报警、PDF、打印等状态。

风险：协议解析、状态管理、UI 回调、存储和调试逻辑互相耦合，修复一个链路容易引入跨模块回归。

建议：优先按边界拆分 BLE 扫描/连接、GATT 写队列、协议编解码、采样流、报警评估、PDF 数据准备、PDF 渲染、文件输出、Room Repository。

### 2. 依赖注入名义存在，实际大量全局单例

证据：

- `BlueToothKitManager`、`LocalStorageKitManager`、`DatabaseBackupHelperManager`、`PrintProtocalKitManager` 都持有全局对象。
- `CapnoEasyApplication` 暴露静态 `context`。
- `BaseActivity` 初始化蓝牙、打印、存储等共享能力。

风险：Activity/Context 泄漏、重复初始化、生命周期不可控、测试难以隔离。

建议：把应用级能力放入 Hilt module；Activity 只消费 ViewModel/UseCase；禁止 ViewModel 或 manager 持有 Activity View 实例。

### 3. 状态模型过大且混入 UI 引用

证据：

- `AppStateModel` 管理患者信息、记录、波形、Toast、Loading、Confirm、导航、PDF 配置、报警阈值等。
- `AppStateModel` 存在 `lineChart: LineChart?` 这类 UI 控件引用。

风险：状态污染跨页面传播；Compose 状态、传统 View 控件和业务数据混用；测试和重构成本高。

建议：拆出 `DeviceState`、`RecordingState`、`PatientState`、`OverlayState`、`PdfExportState`、`SettingsState`；UI 控件引用改为数据输入/输出事件。

### 4. 异步任务和生命周期绑定不足

证据：

- `SearchActivity.kt` 使用 `GlobalScope.launch(Dispatchers.Main)`。
- `HotmeltPinter.kt` 使用 `GlobalScope.launch` 处理打印波形。
- `PDFKit.kt` 使用已废弃 `AsyncTask`，并通过 `Handler` + `CountDownLatch` 等待主线程 PDF 生成。
- `DatabaseBackupHelperKit.kt` 启动路径使用 `runBlocking`。

风险：页面销毁后后台任务继续写 UI/设备；启动阻塞；PDF/备份/打印无法可靠取消；异常路径只打印堆栈。

建议：迁移到 `lifecycleScope`、`viewModelScope` 或明确 application scope；PDF/备份/打印返回结构化结果并支持取消；禁止启动链路阻塞主线程。

### 5. BLE 协议层缺少显式状态机

证据：

- `BlueToothKit` 内部共享 `receivedArray`、`sendArray`、GATT 特征、设备状态、报警状态。
- 收包后启动临时协程解析，帧缓存、校验、状态更新耦合在同一类。
- 设备信息查询存在 `while(true)` 轮询。

风险：断连、权限变化、半连接、帧错位、设备不支持命令时行为不可预测。

建议：建立 BLE 状态机：`Idle -> Scanning -> Connecting -> DiscoveringServices -> Authenticating -> Streaming -> Disconnecting/Error`；协议编解码提取为纯 Kotlin 模块并补测试向量。

### 6. 权限和发布合规策略偏粗

证据：

- Android Manifest 声明后台定位、全文件管理、旧存储权限。
- `MainActivity.onCreate()` 直接跳转全文件管理权限设置。
- `allowBackup=true`，备份规则仍是模板内容。
- Bugly debug 标志存在发布风险。

风险：Android 11+ 存储策略、应用商店审核、医疗/患者数据备份合规和首次启动体验都有风险。

建议：按真实功能拆权限；移除无用权限；备份策略单独产品化；生产包关闭 debug 标志；对患者数据明确 exclude/include 规则。

### 7. 数据库备份/恢复链路半完成

证据：

- `CapnoEasyApplication` 启动时直接调用备份 helper，但相关 TODO 仍未闭环。
- `DatabaseBackupHelperKit` 会关闭 Room 后复制 DB/WAL/SHM，恢复流程强依赖外部文件存在。
- Room `version = 2` 但 `exportSchema = false`，缺少 v2 schema 可追踪性。

风险：首次安装、升级、卸载重装、权限拒绝、文件缺失时可能丢数据、恢复失败或阻塞启动。

建议：备份失败不得阻塞启动；恢复必须有用户确认、事务边界和失败回滚；开启 Room schema 导出并补 migration 测试。

## 中风险问题

### 8. iOS 与 Android 是平行实现，缺少共享协议基准

证据：

- Android BLE 协议在 `BlueToothKit.kt`，iOS 协议在 `BluetoothManage.swift`。
- 两端各自处理报警、单位、设备状态、PDF 导出。
- iOS 历史数据主要依赖 `BluetoothManager.totalCO2WavedData` 当前内存数据，Android 使用 Room chunk 持久化。

风险：两端行为漂移；协议修复需要人工同步；报警和 PDF 规则容易不一致。

建议：维护跨平台协议测试向量：输入字节帧、期望 CO2/RR/报警/设备状态；Android/iOS 各自用同一组样例验证。

### 9. iOS 管理类和文案系统过大

证据：

- `BluetoothManager` 同时负责 CoreBluetooth、协议解析、设备配置、报警音频、采样历史。
- `AppConfigManage.getTextByKey()` 通过长 switch 管理中英文文案。
- `AlertConfigView.RangeSlider` TODO 指出滑块标签遮挡、左右滑块可交叉。

风险：UI、协议、配置和国际化互相耦合；扩展语言、调整报警 UI 或修改协议时容易牵连。

建议：使用标准 localization；拆分 `BluetoothTransport`、`SensorProtocolParser`、`AlarmEvaluator`、`DeviceSettingsStore`。

### 10. 构建依赖和模块边界不干净

证据：

- Android app 依赖存在重复声明。
- `hotmeltprint` 重复声明 `mpandroidchart`，同时通过 fileTree 和 files 引入同一 jar。
- `hotmeltprint` 作为库模块依赖打印 SDK、图表、iText、Hilt，职责偏重。

风险：依赖冲突难定位；AGP/Kotlin/Gradle 升级成本高；模块复用边界不清晰。

建议：迁移到 version catalog 统一依赖；删除重复和未使用依赖；打印 SDK 封装成窄接口，避免业务层感知 SDK 细节。

### 11. 生产调试输出较多

证据：

- Android 多处 `println("wswTest...")`、`printStackTrace()`。
- iOS 多处 `print(...)` 输出设备、PDF、协议校验信息。

风险：日志噪声影响定位；可能暴露患者数据、设备状态、文件路径。

建议：建立统一 logger；按 debug/release 控制输出；异常使用结构化错误码和用户可见错误。

### 12. 测试覆盖不足

证据：

- Android `app` 和 `hotmeltprint` 的 test/androidTest 主要是模板测试。
- iOS XCTest/UI Test 也是模板内容。
- BLE 协议解析、Room migration、PDF 分页、报警阈值、权限流程没有自动化回归保护。

风险：当前架构下任何拆分都缺少安全网。

建议：先补纯逻辑测试，再做结构拆分。优先级：BLE 帧解析、报警边界、Room migration、PDF 分段/时间轴、iOS 协议解析。

## 建议治理顺序

1. 建立最小安全网：BLE 协议解析测试、Room migration 测试、PDF 分段/时间轴测试。
2. Android 拆 `BlueToothKit`：先提纯协议解析和状态机，不同时改 UI。
3. 收敛 `AppStateModel`：按业务域拆状态，移除 UI 控件引用。
4. 重做备份/恢复策略：去掉启动阻塞和全文件权限强依赖。
5. 建立跨平台协议样例：Android/iOS 共用同一组协议输入和期望输出。
6. 清理发布风险：Manifest 权限、backup rules、Bugly debug、生产日志。
7. 拆 PDF 和打印：数据准备、渲染、输出、SDK 调用分层。
8. 清理依赖：删除重复依赖，收窄模块 API，记录本地 jar 来源和版本。

## 判断标准

完成治理后应满足：

- BLE 协议解析可脱离 Android/iOS UI 单独测试。
- UI 状态不持有 View/Activity/LineChart。
- 备份失败不会影响启动；恢复有确认、回滚和错误反馈。
- Android/iOS 对同一协议样例输出一致。
- PDF 生成有固定输入回归测试。
- Manifest 权限能逐条解释其业务必要性。
- 核心模块改动有自动化测试覆盖，而不是依赖真机手测。
