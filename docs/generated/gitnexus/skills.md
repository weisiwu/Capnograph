# 上下文感知开发技能模块

## 概述

`context-aware-dev` 是一个面向代码仓库的上下文感知开发流程技能模块。它定义了一套标准化的 Agent 工作流，确保在回答、调试、实现或评审代码时，始终基于项目实体、ID、配置、文档和历史上下文进行操作，并在任务完成后同步上下文文档。

该模块的核心价值在于解决 LLM 在大型代码仓库中常见的“上下文漂移”问题——通过强制性的两阶段工具流和上下文同步机制，保证 Agent 的每一次操作都锚定在项目实际状态上。

### 关键组件

#### 1. 上下文定位文件

模块定义了一组必须优先读取的上下文文件，按优先级排列：

| 文件路径 | 用途 |
|---------|------|
| `AGENTS.md` | 项目根目录，告知 Agent 应使用两阶段工具流 |
| `context/实体标识映射.md` | 实体到 ID 的映射关系 |
| `context/*映射*.md` | 其他映射文件 |
| `context/项目上下文.md` | 项目整体上下文 |
| `context/项目记忆.md` | 项目历史记忆 |
| `docs/` | 补充文档目录 |

#### 2. 实体上下文文档

每个实体对应一个独立的上下文文档，存储在 `context/文档/实体/<id>.md`。模块要求只加载当前请求需要的实体上下文，避免信息过载。

#### 3. 辅助导航工具

- **GitNexus**: 用于代码符号查询、调用方分析、执行流追踪和影响范围检测
- **Graphify**: 用于概念查找、文档密集问题、关系/路径探索

#### 4. context-seed 集成

当项目安装了 `context-seed` 时，模块强制使用两阶段 Agent 工具流：

1. **准备阶段**: 运行 `python -m context_seed prepare . --operation init --json` 或 `python -m context_seed prepare . --operation finalize --commit HEAD --json`
2. **应用阶段**: 填充计划中的 `planned_files[].content` 后运行 `python -m context_seed apply . --plan-file context-seed-plan.json`

## 速度策略

模块支持四种速度层级，影响工具调用频率和上下文加载深度：

| 层级 | 触发关键词 | 行为 |
|------|-----------|------|
| **Fast** | `Fast`、`快速` | 优先定向 `rg` 和章节读取；最多一次图查询 |
| **Standard** | `Standard`、`标准` | 平衡速度和深度 |
| **Deep** | `Deep`、`深度` | 全面加载上下文，多次图查询 |
| **自动升级** | 高风险编辑前 | 当当前层级不足以覆盖共享代码、数据模式、生成资产、用户流程或大范围重构时 |

## 工作流详解

### 步骤 1: 上下文解析

使用定向 `rg`（ripgrep）从上下文定位文件中解析请求涉及的实体与 ID。只解析当前请求需要的实体，避免加载无关信息。

### 步骤 2: 工作集记录

记录一个最小工作集，包含：
- 实体名称
- 唯一 ID
- 所属领域
- 源码文件路径
- 补充上下文路径

### 步骤 3: 上下文加载

优先只加载映射行中 `补充上下文` 指向的 `context/文档/实体/<id>.md` 文件。仅在需要仓库总览时读取 `context/文档/代码库上下文.md`。

### 步骤 4: 工具辅助分析

仅在以下情况使用 GitNexus 或 Graphify：
- 直接定位文档有歧义
- 影响面重要，需要确认

工具使用规则：
- **GitNexus**: `gitnexus query`、`gitnexus impact`、`gitnexus detect-changes` 或 `npx -y gitnexus@latest ...`
- **Graphify**: `graphify query "<entity>" --budget 800`、`graphify explain "<symbol>"` 或 `graphify path "<A>" "<B>"`

### 步骤 5: 源码验证

在编辑前，必须用源码文件验证图工具的输出，确保准确性。

### 步骤 6: 修改与验证

执行最小安全修改，并运行最小有意义的验证。

### 步骤 7: 上下文同步

修改后同步受影响的上下文文档：
- 更新 `context/实体标识映射.md`（当 ID、源码文件或补充文档变化时）
- 更新 `context/文档/实体/<id>.md`（当行为、字段、资产、流程或假设变化时）
- 同步 `docs/` 中的镜像内容

### 步骤 8: 索引刷新

项目文件发生变化时，在结束时刷新一次可选索引：
- GitNexus: `gitnexus analyze --no-stats`
- Graphify: `graphify update . --no-cluster`

### 步骤 9: 提交后收尾

用户完成 `git commit` 后运行：
- Agent 推荐: `python -m context_seed prepare . --operation finalize --commit HEAD --json`
- 人工快捷: `python -m context_seed finalize .`

收尾动作包括分析最近提交，更新 `context/项目上下文.md`、`context/项目记忆.md`、`context/文档/代码库上下文.md` 和实体映射。

## 输出要求

模块要求 Agent 在每次操作后输出：
1. 关键实体或 ID 说明（有助于理解时）
2. 已运行的验证说明
3. 上下文文档更新说明（或明确无需同步）
4. GitNexus 和 Graphify 使用情况，以及索引刷新状态

### 依赖关系

- **可选依赖**: GitNexus、Graphify、context-seed
- **强制依赖**: ripgrep（`rg`）、git

## 安全约束

`context-seed` 的 `apply` 命令只允许目标项目内相对路径，拒绝绝对路径和包含 `..` 的路径，防止路径穿越攻击。

## 最佳实践

1. **始终从 AGENTS.md 开始**: 这是模块的入口点，告知 Agent 应使用两阶段工具流
2. **最小上下文原则**: 只加载当前请求需要的实体上下文，避免信息过载
3. **源码验证优先**: 图工具输出必须用源码验证，避免幻觉
4. **同步一致性**: 修改后必须同步所有受影响的上下文文档，包括 `docs/` 中的镜像内容
5. **速度层级选择**: 未指定时使用 `Fast`，高风险编辑前考虑升级
